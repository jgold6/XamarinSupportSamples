Introduction to Android with Xamarin Studio

Exercise 1: Create a Xamarin.Android application in Xamarin Studio

Knowledge needed for the exercise

Development machine setup
You will need to configure your development machine before you begin coding.
For step-by-step guidance, see Installing Xamarin.Android (for Windows or Mac).
Activity basics
Android apps are built from Activities. Each Activity implements one task and they all work together to provide the app's services.
The first sections of this page provide a nice overview of Activities: Activity Lifecycle.
Launchable activities
By default, your Activities will not appear on the Android Launcher screen. Set the MainLauncher property of the ActivityAttribute to true to specify that an Activity acts as an entry point for your app. The build process transfers this setting to your app's manifest. When your app is installed, Android will read the manifest and add the designated Activities to the Launcher screen.
The syntax is summarized in the "Launchable from Application Chooser" section here: Working with AndroidManifest.xml.
Executing your code
You can test your apps on a physical device or on the Android emulator.
To setup a device, see Set Up Device for Development.
To use the emulator, see Configure the Emulator.

Exercise 2: Use the Xamarin.Android Designer to add controls to a UI

Knowledge needed for the exercise

Using the Designer
Xamarin.Android includes a UI Designer that is integrated into Visual Studio and Xamarin Studio. The Designer behavior is similar in both environments (the documentation shows Xamarin Studio).
To learn more about the Designer, see Designer Basics.
Input controls
Android has an extensive set of standard controls. For this exercise, we will use a text display control, a text input control, and two buttons. In Xamarin.Android, these controls will feel like familiar .NET types: the text controls have a read/write Text property, the buttons have Click events, etc.
To learn the basic XML syntax for the input controls, see Input Controls (Android).

Exercise 3: Implement app behavior in C# code-behind

Knowledge needed for the exercise

Adding code-behind logic
The Activity.OnCreate method is a great place to put initialization logic, such as event subscriptions. Add your code after the call to SetContentView. The controls will be created in SetContentView so you want to wait until after that method completes before you work with them.
Accessing controls
You can access controls from code-behind by using the FindViewById method. You pass in the id of the control you are looking for and it returns a reference to that control. Controls that have an id in the XML will automatically have an id generated for use in code. For example, setting an id of @+id/MyId in XML will create a field named Resource.Id.MyId that you can pass to FindViewById. The Phoneword application uses three ids: PhoneNumberText, TranslateButton, and CallButton.
To view the method documentation, see FindViewById<T> Method.
Subscribing to events
You can subscribe to events in code-behind using standard C# syntax. Traditional methods and lambda expressions are both supported. The Button control has a Click event (actually defined in the View base class).
To view the event documentation, see Click Event.
Text input
You will need to read user input from the Text property of an EditText control and display results in the Text property of a Button control.
To learn how to work with text input, see Capture User Input Text.
Displaying an alert
Use the dialog builder to create an alert. 
var dialog = new AlertDialog.Builder(this);
dialog.SetMessage("My Message");
dialog.SetNeutralButton("My Button Label", 
  delegate
  {
    ...
  });
dialog.SetNegativeButton("Cancel",
  delegate
  {
  });
dialog.Show();
To learn more about dialogs, see Dialogs (Android).
Dialing a phone number in Xamarin.Android
To dial a phone number in Android, you create a Intent that uses the ActionCall action and the tel URI scheme. When you start the intent, Android will search for an installed Activity that can handle your request and start that Activity for you. The Android Emulator cannot place phone calls so the call will not actually connect even though the call UI will be displayed. 
var callIntent = new Intent(Intent.ActionCall);
callIntent.SetData(Android.Net.Uri.Parse("tel:" + translatedNumber));
StartActivity(callIntent);
Adding a manifest permission
To access many of the Android system resources, apps must add a declaration to their manifest. This is done in the Xamarin Studio Project Options; the value is then automatically added to the app's AndroidManifest.xml file. To add the permission:
  1. Open the Project Options in Xamarin Studio.
  2. Select the Android Application section from the left.
  3. Locate the Required permissions list box.
  4. Check the box next to the needed permission.
Hiding the Android soft keyboard
Depending on your choice of execution environment, you may end up running your app on a device without a hardware keyboard or in an emulator configured without a hardware keyboard. In this situation, you can improve the user experience by programmatically hiding the soft keyboard as needed. The code below shows how to hide the soft keyboard. Replace the referenceToMyControl in the code with a reference to the control that caused the soft keyboard to display; for example, an EditText. 
var imm = (InputMethodManager)GetSystemService(Context.InputMethodService);
imm.HideSoftInputFromWindow(referenceToMyControl.WindowToken, 0);
Adding files to Xamarin Studio
  1. Right-click on the Project in the Solution pad and select Add > Add Files...
  2. Navigate to the file; select it, and click Open to add it to the project.

Exercise 4: Code two screens and navigate between them

Knowledge needed for the exercise

Passing data to an activity
One activity can start another using an Intent. The source activity can put key/value pairs of data into the Intent (this data is called the Intent extras). The Intent object will be available inside the target activity in an inherited property named Intent. The target can use this property to retrieve the extras as needed.
See Pass Data Between Activities.
Creating multiple screens
Android apps can have multiple screens. Typically, each screen will be an Activity and you will use an Intent to launch the new activity from the old one. To learn how to define multiple Activities and how to navigate between them.
See Handling Navigation with Xamarin.Android.

Exercise 5: Use Resources to incorporate custom labels and icons

Knowledge needed for the exercise

Icon sizes
Android applications run on a variety of screen sizes and resolutions. You should supply several sizes of each icon to ensure it looks good in every resolution.
Android uses a folder-naming convention to organize icon resources. For example, if your icon file is named icon.png, you will supply one folder named Resources > drawable-mdpi containing icon.png for medium-dpi displays, another folder named Resources > drawable-hdpi also containing an icon.png file for hi-dpi displays, etc.
See Iconography (Android).
Application icon and label
An Android application can have an Icon and a Label. These values will be shown to the user on the Android Settings screen under the Apps category. They also serve as default values for the corresponding Activity attributes.
Note that the application Logo is different from the application Icon. The Logo is used elsewhere, e.g. as the default image for the ActionBar. Here, we are only concerned with the Icon since that is the image the user will see as representative of your app.
These values are packaged into the application manifest. Xamarin gives you a UI in the project properties to set these values so you do not need to edit the manifest directly. Activity class.
See the sections on android:icon and android:label in the App manifest application element (Android).
Activity icon and label
An Android application is a collection of collaborating activities. Activities with the MainLauncher = true designation appear on the Apps screen. An application can have more than one entry-point activity and all of them will appear on the launcher screen.
Each activity can have a Label and an Icon. These are diplayed at the top of the screen when the activity is in the foreground. For activities with the MainLauncher = true designation, the Label and Icon are also used on the Apps screen.
If either of these are not set, they default to the corresponding application-level value.
In Xamarin, these values are set using properties on the ActivityAttribute that you apply to each Activity class.
See the sections on android:icon and android:label in the App manifest activity element (Android).
